1. System Architecture Overview
1.1 Architecture Pattern
Architecture Decision
LIGMA uses an Event-Sourced, WebSocket-first architecture. All state changes are represented as immutable events. The canvas state at any point is derived by replaying events from an append-only log. Real-time communication uses a custom-built WebSocket server — no third-party WebSocket hosting services.

1.2 High-Level Component Diagram
The system has four primary layers:
Browser Client — React SPA (canvas rendering, WebSocket client, AI badge display, panels)
Custom WebSocket Server — Node.js ws-based server managing room sessions, event broadcasting, RBAC enforcement
REST API Server — Express.js handling authentication, room management, initial state delivery, AI classification proxy
Persistence Layer — Database (PostgreSQL or SQLite) storing events, rooms, users, node ACLs

2.2 WebSocket Client Implementation
Critical Constraint
The WebSocket client uses the native browser WebSocket API directly. No third-party WebSocket client libraries (e.g., Socket.io-client, Pusher-js, Ably) are used. Connection management, heartbeat, and reconnection logic are all custom-implemented.

WebSocketClient Class
class WebSocketClient {
  private ws: WebSocket | null = null;
  private roomId: string;
  private token: string;
  private lastSequenceNumber: number = 0;
  private reconnectAttempt: number = 0;
  private maxReconnectAttempts: number = 10;
  private heartbeatInterval: ReturnType&lt;typeof setInterval&gt; | null = null;
  private messageHandlers: Map&lt;string, Function&gt; = new Map();

  connect(): void {
    const wsUrl = `${WS_BASE_URL}/ws?roomId=${this.roomId}&amp;token=${this.token}`;
    this.ws = new WebSocket(wsUrl);
    this.ws.onopen = this.handleOpen.bind(this);
    this.ws.onmessage = this.handleMessage.bind(this);
    this.ws.onclose = this.handleClose.bind(this);
    this.ws.onerror = this.handleError.bind(this);
  }

  private handleOpen(): void {
    this.reconnectAttempt = 0;
    this.startHeartbeat();
    this.send({ type: &apos;RECONNECT&apos;, lastSequenceNumber: this.lastSequenceNumber });
  }

  private handleClose(event: CloseEvent): void {
    this.stopHeartbeat();
    if (!event.wasClean) this.scheduleReconnect();
  }

  private scheduleReconnect(): void {
    if (this.reconnectAttempt &gt;= this.maxReconnectAttempts) return;
    const baseDelay = Math.min(500 * 2 ** this.reconnectAttempt, 30000);
    const jitter = baseDelay * 0.2 * (Math.random() * 2 - 1);
    setTimeout(() =&gt; this.connect(), baseDelay + jitter);
    this.reconnectAttempt++;
  }

  private startHeartbeat(): void {
    this.heartbeatInterval = setInterval(() =&gt; {
      if (this.ws?.readyState === WebSocket.OPEN)
        this.ws.send(JSON.stringify({ type: &apos;PING&apos; }));
    }, 5000);
  }
}

2.3 CRDT Integration (Yjs)
Text conflict resolution for canvas node text content uses Yjs Y.Text CRDT:
import * as Y from &apos;yjs&apos;;

// Per-node Yjs document (one doc per text node)
const ydoc = new Y.Doc();
const ytext = ydoc.getText(&apos;content&apos;);

// Apply remote update (received via WebSocket CANVAS_DELTA with type TEXT_CRDT)
Y.applyUpdate(ydoc, remoteUpdate);

// Encode local update to broadcast
ydoc.on(&apos;update&apos;, (update: Uint8Array) =&gt; {
  wsClient.send({ type: &apos;CANVAS_DELTA&apos;, nodeId, deltaType: &apos;TEXT_CRDT&apos;,
    payload: Array.from(update) });
});

For non-text properties (position x/y, dimensions w/h, color, lock state), the conflict resolution strategy is:
Timestamped last-write-wins: each delta carries a client-generated timestamp; the server assigns a sequenceNumber; clients apply in sequence order
Rationale: position/style conflicts are rare and non-semantic; data loss risk is low; CRDT overhead for positional data is not justified

3.2 Custom WebSocket Server
Implementation Requirement
The WebSocket server is built using the ws npm package on top of Node.js http.createServer. Socket.io, Pusher, Ably, AWS API Gateway WebSockets, and all other third-party managed WebSocket services are STRICTLY PROHIBITED. Every aspect — connection management, room routing, heartbeat, broadcast, reconnection replay — is custom-engineered.

Server Bootstrap
import http from &apos;http&apos;;
import WebSocket, { WebSocketServer } from &apos;ws&apos;;
import express from &apos;express&apos;;

const app = express();
const httpServer = http.createServer(app);
const wss = new WebSocketServer({ noServer: true });

// Upgrade HTTP -&gt; WebSocket
httpServer.on(&apos;upgrade&apos;, (request, socket, head) =&gt; {
  const token = extractTokenFromURL(request.url);
  verifyJWT(token, (err, user) =&gt; {
    if (err) { socket.destroy(); return; }
    wss.handleUpgrade(request, socket, head, (ws) =&gt; {
      wss.emit(&apos;connection&apos;, ws, request, user);
    });
  });
});

Connection Registry
interface ClientConnection {
  ws: WebSocket;
  userId: string;
  roomId: string;
  role: &apos;LEAD&apos; | &apos;CONTRIBUTOR&apos; | &apos;VIEWER&apos;;
  displayName: string;
  color: string;
  lastSequenceNumber: number;
  lastSeen: Date;
}

class RoomRegistry {
  private rooms: Map&lt;string, Set&lt;ClientConnection&gt;&gt; = new Map();

  addClient(roomId: string, client: ClientConnection): void {
    if (!this.rooms.has(roomId)) this.rooms.set(roomId, new Set());
    this.rooms.get(roomId)!.add(client);
  }

  broadcast(roomId: string, message: object, excludeClient?: ClientConnection): void {
    const clients = this.rooms.get(roomId);
    if (!clients) return;
    const payload = JSON.stringify(message);
    for (const client of clients) {
      if (client === excludeClient) continue;
      if (client.ws.readyState === WebSocket.OPEN)
        client.ws.send(payload);
    }
  }
}

Message Protocol — Full Specification
All WebSocket messages are JSON-encoded UTF-8 strings with the following envelope:
{
  type: MessageType;         // string enum
  roomId: string;            // UUID
  userId?: string;           // sender user ID
  sequenceNumber?: number;   // server-assigned, monotonic per room
  timestamp: string;         // ISO 8601
  payload: object;           // message-specific data
}

3.3 RBAC Server-Side Enforcement
class RBACService {
  canMutateNode(client: ClientConnection, node: Node, mutation: string): boolean {
    // Viewers can never mutate
    if (client.role === &apos;VIEWER&apos;) return false;

    // Locked nodes: only Leads can mutate
    if (node.locked &amp;&amp; client.role !== &apos;LEAD&apos;) return false;

    // Lock/unlock operations: Lead only
    if ([&apos;NODE_LOCKED&apos;,&apos;NODE_UNLOCKED&apos;].includes(mutation) &amp;&amp; client.role !== &apos;LEAD&apos;)
      return false;

    return true;
  }
}

The RBAC check runs in canvasDeltaHandler before ANY event is written to the log or broadcast. Unauthorized mutations result in an ERROR message sent only to the offending client:
{ type: &apos;ERROR&apos;, code: &apos;RBAC_DENIED&apos;,
  message: &apos;Permission denied: node is locked or your role is insufficient&apos;,
  nodeId: delta.nodeId }

3.4 Event Store
Event Append Pattern
class EventStore {
  async appendEvent(event: CanvasEvent): Promise&lt;void&gt; {
    // sequenceNumber is DB-assigned via auto-increment
    await db.execute(
      `INSERT INTO events (event_id, room_id, event_type, user_id,
       payload, created_at) VALUES (?, ?, ?, ?, ?, ?)`,
      [event.eventId, event.roomId, event.eventType,
       event.userId, JSON.stringify(event.payload), new Date()]
    );
  }

  async getEventsSince(roomId: string, sequenceNumber: number): Promise&lt;CanvasEvent[]&gt; {
    return db.query(
      `SELECT * FROM events WHERE room_id = ? AND sequence_number &gt; ?
       ORDER BY sequence_number ASC`,
      [roomId, sequenceNumber]
    );
  }
}

3.5 AI Classification Service
class AIClassifierService {
  async classify(text: string): Promise&lt;IntentTag&gt; {
    try {
      const response = await fetch(&apos;https://api.anthropic.com/v1/messages&apos;, {
        method: &apos;POST&apos;,
        headers: { &apos;x-api-key&apos;: process.env.ANTHROPIC_API_KEY,
                   &apos;anthropic-version&apos;: &apos;2023-06-01&apos;,
                   &apos;content-type&apos;: &apos;application/json&apos; },
        body: JSON.stringify({
          model: &apos;claude-haiku-4-5-20251001&apos;,
          max_tokens: 20,
          messages: [{
            role: &apos;user&apos;,
            content: `Classify this text as exactly one of: ACTION_ITEM, DECISION, OPEN_QUESTION, REFERENCE. Respond with only the tag.\n\nText: ${text}`
          }]
        })
      });
      const data = await response.json();
      const tag = data.content[0]?.text?.trim();
      return VALID_TAGS.includes(tag) ? tag : &apos;UNCLASSIFIED&apos;;
    } catch {
      return this.fallbackClassify(text);
    }
  }

  private fallbackClassify(text: string): IntentTag {
    const lower = text.toLowerCase();
    if (/^(todo:|action:|assign to|@\w+:)/.test(lower)) return &apos;ACTION_ITEM&apos;;
    if (/^(decided:|we will|agreed:|resolution:)/.test(lower)) return &apos;DECISION&apos;;
    if (/\?|^(how|why|what|should|when|where)/.test(lower)) return &apos;OPEN_QUESTION&apos;;
    return &apos;REFERENCE&apos;;
  }
}

  joined_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),

-- Append-only event log
CREATE TABLE events (
  sequence_number  BIGSERIAL PRIMARY KEY,          -- auto-increment, per-table global
  event_id         UUID NOT NULL UNIQUE,
  room_id          UUID NOT NULL REFERENCES rooms(room_id),
  event_type       VARCHAR(50) NOT NULL,
  user_id          UUID REFERENCES users(user_id),
  payload          JSONB NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_events_room_seq ON events(room_id, sequence_number);

-- Current node state cache (derived from events, for fast initial load)
CREATE TABLE canvas_nodes (
  node_id       UUID PRIMARY KEY,
  room_id       UUID NOT NULL REFERENCES rooms(room_id),
  node_type     VARCHAR(30) NOT NULL,
  x             FLOAT NOT NULL,
  y             FLOAT NOT NULL,
  width         FLOAT,
  height        FLOAT,
  text_content  TEXT,
  crdt_state    BYTEA,            -- serialized Yjs document state
  style         JSONB,
  author_id     UUID REFERENCES users(user_id),
  ai_tag        VARCHAR(30),
  locked        BOOLEAN NOT NULL DEFAULT FALSE,
  locked_by     UUID REFERENCES users(user_id),
  z_index       INTEGER NOT NULL DEFAULT 0,
  deleted_at    TIMESTAMPTZ,       -- soft delete
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_nodes_room ON canvas_nodes(room_id, deleted_at);

-- Task Board entries
CREATE TABLE tasks (
  task_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id       UUID NOT NULL REFERENCES rooms(room_id),
  node_id       UUID NOT NULL REFERENCES canvas_nodes(node_id),
  text_content  TEXT NOT NULL,
  author_id     UUID REFERENCES users(user_id),
  status        VARCHAR(20) NOT NULL DEFAULT &apos;ACTIVE&apos;,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

4.2 Schema Notes
canvas_nodes is a materialized cache derived from the event log. It is updated as events are processed (write-through). It is never the source of truth — the events table is.
crdt_state stores the serialized Yjs document binary (encodeStateAsUpdate(ydoc)) for each text node. On connect, the client receives this binary and initializes its local Yjs doc from it.
sequence_number is a BIGSERIAL auto-increment across all rooms (global sequence). For per-room replay, filter by room_id AND sequence_number &gt; lastKnown.
deleted_at: soft-delete pattern ensures node history is preserved; hard delete is never performed.

6. Security Architecture
6.1 Authentication Flow
Client calls POST /api/auth/join with displayName + optional roomId
Server creates user record, assigns cursor color, issues JWT (HS256 signed with server secret)
JWT payload: { userId, roomId, role, exp }
Client stores JWT in localStorage (acceptable for hackathon; sessionStorage for higher security)
All REST requests include: Authorization: Bearer &lt;token&gt;
WebSocket connect URL: wss://host/ws?roomId=X&amp;token=&lt;token&gt;
Server verifies JWT before upgrade; rejects with 401 if invalid

6.2 Input Sanitization
All text content in canvas nodes is sanitized server-side using DOMPurify (server-side) or equivalent before storage
Node payload schemas are validated using Zod or Joi before processing
Maximum text length enforced: 10,000 characters per node (server rejects oversized payloads with ERROR message)
roomId and nodeId validated as UUID format before any database query
Rate limiting on REST: 100 requests/minute per IP using express-rate-limit
Rate limiting on WebSocket: 100 messages/second per connection; excess messages dropped with ERROR event

6.3 RBAC Enforcement Layers

7. Performance &amp; Scalability
7.1 WebSocket Optimizations
Cursor move messages are throttled client-side to 50ms intervals using requestAnimationFrame-based scheduler
Canvas delta messages are batched: if multiple deltas occur within 16ms (one frame), they are sent as a single array payload
Full state snapshots are paginated into chunks of 100 nodes to prevent large initial payload blocking
The server broadcasts delta payloads — NOT full state — ensuring message sizes remain small (typical delta: 200–500 bytes)

7.2 Canvas Rendering Performance
Canvas nodes are rendered in a Konva Layer; only dirty nodes trigger re-render (not full canvas redraw)
Remote cursor positions are interpolated via requestAnimationFrame loop, not triggered by message arrival
Event log panel uses virtual scrolling (react-virtual or react-window) to handle 10,000+ events without DOM bloat
Task Board uses virtual list rendering for large task counts
Canvas transforms (pan/zoom) use CSS transform on the container where possible for GPU acceleration

7.3 Scalability Notes
Hackathon Scope Note
For the hackathon, single-process WebSocket server is sufficient. The architecture is designed to allow horizontal scaling via Redis pub/sub (replacing in-process RoomRegistry with Redis channels) without changing the message protocol. This should be noted in the README as a post-hackathon scaling path.

8.2 Optimistic UI Pattern
All canvas mutations are applied optimistically (immediately on the local client) before confirmation from the server. If the server returns an error:
The client identifies the rejected mutation by correlating a client-generated requestId
The optimistic update is reverted (previous node state is restored from local snapshot)
An error toast is shown with the reason
The canvas is considered potentially stale; a silent background state refresh is triggered

14. Authentication &amp; Authorization — Full Specification
14.1 Authentication Flow (Detailed)
User submits POST /api/auth/join with { displayName: string, roomId?: string, role?: &apos;LEAD&apos;|&apos;CONTRIBUTOR&apos;|&apos;VIEWER&apos; }
Server validates displayName (1–100 chars, sanitized); defaults role to CONTRIBUTOR if unspecified
Server creates user record in users table; assigns cursor color (round-robin from 10-color palette)
If roomId provided and room exists: server creates room_members entry; if room not found: 404 returned
If no roomId: server creates new room, assigns LEAD role to this user
Server signs JWT with HS256 algorithm: payload = { userId, roomId, role, displayName, color, iat, exp }
JWT expiry: 3600 seconds (1 hour) — [ASSUMPTION] sufficient for hackathon demo sessions
Server returns full session object (see Section 5.1)
Client stores JWT in localStorage key &apos;ligma_token&apos;; roomId in localStorage key &apos;ligma_room&apos;

14.2 Token Lifecycle &amp; Refresh
[ASSUMPTION] Token Refresh Strategy
For the hackathon MVP, token refresh is handled by redirect: when AUTH_EXPIRED error is received via WebSocket, the client redirects to /login. A silent refresh endpoint (POST /api/auth/refresh) is documented here as a post-hackathon improvement.

Token expiry is checked on every REST request via authMiddleware.ts

14.4 Security Boundary: Client vs Server
Client-side RBAC is for UX only (hiding/disabling controls). Server-side RBAC is the security boundary. The following server-side checks are mandatory:
Every CANVAS_DELTA WebSocket message: RBACService.canMutateNode() called before event write
Every PATCH /api/rooms/:id/members: role check in route handler before DB update
JWT signature verified on every request/connection — no trust without valid signature
nodeId existence verified against DB before processing any delta to prevent ghost node attacks

19. Bonus Features — Technical Specification
19.1 Presence Heatmap
Implementation Approach
The heatmap is a canvas overlay derived from the event log. Edit density per canvas coordinate region is calculated server-side from NODE_MOVED, NODE_TEXT_UPDATED, NODE_CREATED events and delivered as a 2D grid of intensity values.

Data Model
interface HeatmapCell {
  gridX: number;     // canvas coordinate bucket (every 100px)
  gridY: number;
  intensity: number; // 0.0 – 1.0 (normalized event count)
}

Generation Algorithm
Server queries events table: SELECT payload-&gt;&gt;&apos;x&apos;, payload-&gt;&gt;&apos;y&apos;, event_type FROM events WHERE room_id = ? AND event_type IN (&apos;NODE_CREATED&apos;,&apos;NODE_MOVED&apos;,&apos;NODE_TEXT_UPDATED&apos;)
Each event coordinate is bucketed into 100px grid cells
Cell hit counts normalized to 0.0–1.0 range
Delivered via GET /api/rooms/:id/heatmap as { cells: HeatmapCell[] }

Frontend Rendering
Heatmap rendered as a semi-transparent Konva Layer above the canvas content layer
Each cell rendered as a colored rectangle: intensity 0 = transparent, 1 = red (#E53E3E) at 50% opacity
Color scale: blue (low) → yellow (medium) → red (high) using linear interpolation
Toggle via keyboard shortcut H; button in toolbar
Heatmap refreshes every 30 seconds when visible (rolling GET /heatmap call)

19.2 Time-Travel Replay
Implementation Approach
Time-travel replay reconstructs canvas state by replaying events up to a selected timestamp. All events from the events table are fetched and applied sequentially in a read-only canvas mode.

Replay State Machine
type ReplayState = &apos;LIVE&apos; | &apos;REPLAY_LOADING&apos; | &apos;REPLAYING&apos; | &apos;PAUSED&apos;;

LIVE → REPLAY_LOADING: user clicks &apos;Time Travel&apos; button
REPLAY_LOADING → REPLAYING: all events fetched from server; initial empty canvas state set
REPLAYING: scrubber position determines which events are applied; events replayed up to scrubber timestamp
Playback speeds: 1x, 2x, 5x, 10x — implemented via setTimeout interval scaling
PAUSED: scrubber stopped; canvas frozen at replay state
Exit replay: button click or pressing Escape → WebSocket reconnect → back to LIVE

Technical Constraints
During replay: WebSocket incoming events are queued but NOT applied to canvas
Replay uses a separate in-memory canvas state (replayState) — does not mutate live canvasStore
All editing tools are disabled during replay; toolbar shows &apos;Replay Mode&apos; banner
Replay canvas is read-only: clicks, drags, keyboard shortcuts do nothing
